## Problem Statement

The "Successful Login After Brute Force" analytics
rule correctly detected a confirmed credential
compromise — 5+ failed logon attempts followed by
a successful authentication from the same source
IP. The rule fired as designed and created
Incident #295 with 11 correlated events and 1
alert. However, the incident's Entities panel
displayed "No entities found" despite the
underlying query output containing a clear
`IpAddress` and `TargetAccount` for every
matching row.

---

## Evidence

**Incident detail panel:**

Evidence:    11 Events, 1 Alert, 0 Bookmarks

Entities (0): -

**Underlying query output columns (confirmed via
Logs blade test run):**

TimeGenerated | IpAddress | TargetAccount | FailCount

The query was returning exactly the data needed
to populate entities — IP address and account
name — but the incident showed none.

---

## Root Cause Analysis

**Primary Root Cause:**

Microsoft Sentinel analytics rules do not
automatically infer entity types from query
output column names. Even when a query returns
a column literally named `IpAddress`, Sentinel
will not treat it as an IP entity unless the rule
is explicitly configured to do so under the
"Entity mapping" section of the rule's "Set rule
logic" tab.

This is a configuration step that is separate
from the query logic itself. A rule author can
write a fully correct, well-tested KQL query and
still produce incidents with zero entity context
if this step is skipped during rule creation.

**Contributing Factors:**

Factor 1: Entity mapping defaults to collapsed/

unconfigured in the rule wizard UI

It is easy to move past this section

without noticing it requires manual setup
Factor 2: The query itself runs and returns correct

results in the Logs/test pane, creating

false confidence that the rule is fully

configured — there is no warning that

entity mapping is missing
Factor 3: This rule was built and validated for

query logic correctness first, with

entity mapping treated as a follow-up

step rather than part of initial setup

## Resolution

**Entity mapping configured on the rule:**

Entity Type: IP

Identifier: Address

Column:     IpAddress
Entity Type: Account

Identifier: Name

Column:     TargetAccount

This was completed via:

Analytics → [Rule Name] → Edit → Set rule logic

→ Alert enhancement → Entity mapping

→ Add new entity (x2) → Save

Future incidents generated by this rule will now
display the correlated IP address and account name
directly in the Entities panel, enabling:

directly in the Entities panel, enabling:

→ Investigation graph functionality

→ Cross-incident entity correlation

## Lessons Learned

Lesson 1: Query correctness ≠ rule completeness

A KQL query that returns the right data in the

test pane does not guarantee the resulting

incident will be operationally useful. Entity

mapping is a required configuration step, not

an optional refinement.
Lesson 2: Validate the Entities panel, not just

the query results

After deploying any new analytics rule, the

first incident it generates should be manually

reviewed for entity population — not just

alert volume and severity.
Lesson 3: This gap is easy to miss precisely

because the rule "works"

The rule fired, the severity was correct, the

event count was correct. Everything LOOKED

successful. The gap only became visible when

manually inspecting the incident detail page —

reinforcing the value of always reviewing full

incident details rather than relying on the

incident list summary view alone.
Lesson 4: Standardize entity mapping as a

checklist item for all custom rules

Going forward, any custom analytics rule

deployment in this lab includes entity mapping

configuration as a mandatory step before the

rule is considered complete.

## Standard Entity Mapping Reference

For future rule development, common entity
mappings for authentication-based detections:

IP Address columns     → Entity Type: IP

Identifier: Address
Account/Username       → Entity Type: Account

columns                   Identifier: Name
Hostname/Computer       → Entity Type: Host

columns                   Identifier: FullName or HostName
Process/Hash columns   → Entity Type: File or Process

Identifier: FileHash or

ProcessId, depending on context
